# DWSS AI Agent System MVP

面向香港政府工程项目 DWSS 的 AI Agent 驱动数字化监管系统原型。

系统服务于顾问、承建商、工程队和项目管理人员，目标是把传统纸质表单、现场巡检、安全记录、质量检查、多角色审批和政府合规要求，转化为可追踪、可审计、可智能辅助的数字工作流系统。

## 项目定位

DWSS 项目的核心难点是工程监管资料复杂、流程长、角色多、合规要求高。项目资料通常包含大量英文政府技术通告、PPT、PDF、Excel 模板、表单样本和客户业务文件，不同文件之间存在交叉要求。

产品经理、开发人员和测试人员需要人工阅读这些资料，再整理出 PRD、workflow、字段清单、角色权限、测试用例和客户沟通问题。这个过程效率低，并且容易遗漏关键合规要求。

本项目希望通过多 Agent 协作，把业务资料理解、流程推理、产品需求生成、测试用例生成和开发任务拆分连接成一个完整闭环。

## 核心痛点

### 文件理解成本高

DWSS 资料大量使用英文工程、政府监管和合规术语，非专业人员很难快速理解每个表单的用途、字段要求和审批规则。

### 工作流复杂

不同表单涉及创建人、填写人、签署人、审核人、责任人和最终确认人。流程中还存在指派、退回、重新提交、完成等状态变化，人工拆解容易出错。

### 合规检查困难

政府工程记录通常要求保留当次现场事实，例如日期、时间、地点、检查人员、责任人、现场照片、整改说明、签名等。人工检查每份记录是否完整，成本高且不稳定。

### 开发交付成本高

业务文件需要转化成开发可以理解的模块、字段、接口、状态机、权限矩阵和验收标准。如果没有 AI 辅助，需求分析和开发沟通会消耗大量时间。

### 多语言沟通成本高

客户、工程团队和开发团队可能使用中文、英文和粤语沟通，容易出现理解偏差。系统需要通过 AI 助手帮助统一业务表达和技术表达。

## 核心逻辑流

本项目不是简单的单轮问答系统，而是围绕下面的长链工作流运行：

```text
文件输入
  -> AI 解析
  -> 业务建模
  -> 工作流生成
  -> 表单执行
  -> 合规检查
  -> 开发与测试输出
```

### 1. 文件解析 Agent

读取政府文件、项目 PPT、Excel 表格、表单模板和客户说明，自动提取项目背景、表单类型、字段要求、角色定义、审批节点、合规要求和潜在风险点。

### 2. 业务理解 Agent

对多个文件进行交叉比对，判断不同文件之间是否存在重复、冲突或补充关系。

例如某个表单在 PPT 中只描述了流程，但在 Excel 模板中体现了具体字段，Agent 需要把两部分信息合并成完整业务规则。

### 3. Workflow Agent

根据文件解析结果生成每个表单的工作流，包括：

- 由谁创建
- 由谁填写
- 需要填写哪些字段
- 由谁签名
- 下一步指派给谁
- 什么情况下退回
- 什么情况下完成

这个过程需要长链推理，因为它不是简单提取文字，而是要把多个角色、表单状态和审批动作组合成可执行流程。

### 4. PRD Agent

将业务规则转化为开发可执行的产品需求文档，包括模块划分、页面结构、字段定义、权限控制、数据模型、状态机、接口需求和验收标准，方便后续在 Codex 中进行代码开发。

### 5. QA Agent

根据 workflow 和 PRD 自动生成测试账号、测试流程、正向场景、异常场景、边界条件和回归测试用例。

示例场景包括缺少签名、缺少现场照片、责任人为空、无权限审批、退回后重新提交等。

### 6. 合规检查 Agent

在用户填写或提交表单时进行自动检查，判断记录是否包含必要的日期、时间、地点、人员、现场照片、整改说明和签名。

如果发现缺失内容，系统会提示用户补充，避免不完整记录进入正式归档。

### 7. 开发协作 Agent

把 PRD、workflow 和测试用例进一步拆解成 Codex 可执行的开发任务，例如数据库表设计、API 接口、前端页面、权限逻辑、状态流转、测试脚本等。

## 多 Agent 协作架构

系统核心是多个 Agent 围绕同一项目资料持续推理：

- 文件解析 Agent：理解资料、提取字段、角色、流程和合规要求
- 业务理解 Agent：跨文件比对，合并重复信息，发现冲突和缺口
- Workflow Agent：生成可执行的表单状态流转和审批流程
- PRD Agent：输出开发可执行的产品需求文档
- QA Agent：生成测试路径、异常场景和回归测试用例
- 合规检查 Agent：在提交前检查记录完整性
- 开发协作 Agent：拆解数据库、API、前端、权限和测试任务

多个 Agent 之间共享上下文，并围绕同一项目资料形成从业务理解到开发落地的完整 AI 工作流。

## 当前 MVP 功能

- 登录认证
- 用户角色
- 项目管理
- 文档录入
- AI Agent：文件解析、Workflow、PRD、QA、合规检查
- 表单记录
- 表单合规检查
- SQLite 数据库
- React 前端
- OpenAI API 接入预留
- Mock Agent：没有 API Key 也能跑通

## 技术栈

- Backend：FastAPI, SQLAlchemy, SQLite
- Frontend：React, Vite, Tailwind CSS
- AI：OpenAI API 或本地 mock agent
- Auth：JWT bearer token

## 后端启动

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
uvicorn main:app --reload --port 8000
```

打开 API 文档：

```text
http://localhost:8000/docs
```

## 前端启动

```bash
cd frontend
npm install
npm run dev
```

打开：

```text
http://localhost:5173
```

## 使用真实 OpenAI

编辑 `backend/.env`：

```env
AI_PROVIDER=openai
OPENAI_API_KEY=你的key
OPENAI_MODEL=gpt-4.1-mini
```

默认是：

```env
AI_PROVIDER=mock
```

## 产品原型目标

由于项目涉及长上下文文件解析、多文档交叉比对、复杂流程推理、多 Agent 协作、PRD 生成、测试用例生成和 Codex 开发任务拆分，token 使用量会比较高。

系统预计每天用于解析项目资料、校验 workflow、整理客户沟通问题、生成开发任务和辅助代码实现。长期目标是从单个项目交付工具，沉淀为面向工程监管、政府项目合规和数字化工作流的 AI Agent 产品原型。
